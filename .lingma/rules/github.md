# GitHub 提交规范

## 1. 提交信息格式
- **标题行**：简短描述变更内容（不超过50个字符）
- **空行**：标题与正文之间必须有空行
- **正文**：详细说明变更原因、实现方式和影响范围（可选）
- **尾部**：关联的Issue编号或其他元数据（可选）

## 2. 提交类型前缀（Conventional Commits）
- `feat:` 新功能
- `fix:` 修复bug
- `docs:` 文档更新
- `style:` 代码格式调整（不影响功能）
- `refactor:` 重构代码（不增加新功能也不修复bug）
- `test:` 添加或修改测试用例
- `chore:` 构建过程或辅助工具的变动（如依赖升级）
- `perf:` 性能优化相关更改
- `ci:` CI/CD配置文件和脚本的更改
- `revert:` 回滚之前的提交
- `build:` 影响构建系统或外部依赖的更改（如gulp、npm等）

## 3. 提交粒度原则
- **原子性提交**：每个提交只包含一个逻辑变更，避免将多个无关改动合并到一个提交中。
- **按模块拆分**：不同模块的修改应分开提交，例如UI层改动和业务逻辑改动分离。
- **及时提交**：完成一个小功能或修复后立即提交，不要累积大量改动后再一次性提交。
- **清晰边界**：确保每次提交的改动有明确的目的和边界，便于后续审查和回溯。

## 4. 中文提交示例
```
feat: 添加用户登录页面组件

- 实现用户名密码输入框验证逻辑
- 集成JWT认证接口调用功能
- 添加加载状态提示和错误处理机制
```

```
fix: 修复媒体卡片图片加载失败问题

- 修正图片URL拼接逻辑中的路径错误
- 增加默认占位图显示机制
- 优化图片懒加载性能表现
```

## 5. 分支命名规范（补充）
- `feature/xxx` - 新功能开发分支
- `fix/xxx` - bug修复分支
- `hotfix/xxx` - 紧急修复分支
- `release/xxx` - 发布准备分支
